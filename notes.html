Q3 — AI vs Pure Code for Zone Detection
Do we need Claude Opus ($15/$75 per 1M tokens) for zone detection, or can pure code do it? What's the guarantee zones are correct without AI seeing the chart?

First: What does the AI actually do right now?

It looks at a KLineChart screenshot and identifies Supply/Demand zones visually. That's it. No reasoning, no strategy decisions — just "here's a zone at $X–$Y on the 4H."


The real cost at scale:

  • 14 coins × 7 timeframes = 98 chart images per scan
  • Scans run every 4H = 6×/day
  • Each image ≈ 1,600 input tokens + ~200 output tokens
  • Input cost: ~$14/day = ~$420/month
  • Output cost: ~$9/day = ~$270/month
  • Total: ~$700/month just to scan zones
  • At 50 users × $500 avg deposit = $25k AUM → your cut is ~$400/month. AI costs more than you earn.
  • Add more coins or strategies and it compounds fast.

Can pure code identify S/D zones correctly?

Yes. This is a solved problem in quant finance. S/D zones have exact mathematical definitions:

  • Demand zone = consolidation base (small candles) immediately before a large bullish impulse candle
  • Supply zone = consolidation base before a large bearish impulse candle
  • Zone boundary = high/low of the base candles (bodies, not wicks)
  • Fresh zone = price has not returned to it since the impulse
  • FVG = gap between candle[i-1].high and candle[i+1].low — pure math
  • Equal highs/lows = within 0.1% tolerance — pure math
  • Volume confirmation = volume at zone retest vs impulse volume — pure math

All of Sonum's rules are written down. Every rule can be a line of code. OHLCV data from Binance gives you everything you need.


The one real advantage of AI vision:

Chart patterns — flags, cups, W-bottoms. These are harder to code precisely. But they are confirmation signals in the strategy, not entry triggers. You can either:

  • Code them (doable — flag = descending channel after an impulse, W-bottom = two equal lows with a bounce)
  • Skip them and rely on zone + gate rules only (still valid — patterns are bonus confirmation)

The real risk of AI that already happened:

  • Apr 18 — Sonnet 4.6 called a 4H zone "1D confirmed" (wrong). Led to overstated confidence and a bad trade.
  • AI is non-deterministic. Same chart, different day = potentially different zone boundaries.
  • You CANNOT build a reliable business on a foundation that gives different answers to the same question.


Three paths — direct comparison:

Path Cost Reliability Build effort
Pure code (OHLCV algo) $0 Deterministic. Same rules every time. 3–5 days
Claude Opus only (current) ~$700/month Non-deterministic. Already mislabelled zones. Already built
Hybrid — code does zones, AI only for pattern confirmation on A-grade setups ~$30–50/month Best of both. Code gates are strict, AI is advisory only. 4–6 days

Recommendation: Pure code for zones + gates. Zero AI in the execution path.

  • Write zone detection as pure Python on OHLCV data (pandas, ~300 lines)
  • All 5 gates = pure code checkmarks — BTC 1D gate, 4H gate, freshness, volume, absorption
  • If ALL gates pass → fire trade. No model in the loop, no hallucination possible.
  • AI stays useful for: backtesting research, reviewing new strategies, one-off analysis. NOT in live execution.
  • This also means the agent runs on EC2 with zero ongoing AI cost. Scales to 1000 coins for free.

The guarantee question answered: Pure code gives you the same guarantee as any rule-based system — if the rules are correct, execution is correct 100% of the time. The strategy rules ARE written down. Code them once, test them, trust them. That's more reliable than asking a model to read a chart image.

How do we run demo trading with minimal slippage gap vs real trading, and what's the go-live criteria?

The 3 main slippage sources for this strategy:

Source Impact How bad
Entry slippage (market order at zone) Price moves while order processes 0.05–0.3% on alts
Zone boundary imprecision Mark zone at $1.392, real entry fills at $1.395 Eats into TP%
SL body-close delay 30min candle closes, order sent, price moves further 0.1–0.5% on volatile candles

Demo platform: Binance Futures Testnet (best option)

  • URL: testnet.binancefuture.com — real order book mechanics, real spread, real latency, fake USDT
  • Fills behave exactly like live. Slippage is real. Closest to production with zero risk.
  • Agent code needs almost zero changes — swap base URL + testnet API keys only
  • Second option: Bybit / MEXC demo mode — uses real live order book, paper money. Better than pure paper, slightly worse than testnet.
  • Do NOT build a custom paper engine — testnet already eliminates simulation approximation error

4 rules to minimize slippage in the agent:

1. Limit orders for entry (not market)

Place limit buy at zone top (demand) or zone bottom (supply). Zero slippage — you're the maker. If price blows through without filling → missed entry → correct per strategy rules (no late entries).

2. Widen zone boundary by 0.1% in agent

Real fills rarely happen at the exact candle boundary. Mark entry as zone_top + 0.1% for longs, zone_bottom - 0.1% for shorts. Matches real-world fill behavior.

3. Simulate realistic slippage per coin in agent config

SLIPPAGE = {
    'BTC': 0.02,      # deep book
    'ETH': 0.03,
    'SOL': 0.05,
    'SUI': 0.08,
    'MOODENG': 0.15,  # thin book
    'FARTCOIN': 0.15,
}

# LONG entry at zone_top $1.392:
fill_price = zone_top * (1 + SLIPPAGE[coin] / 100)
# fills at $1.3933, not $1.3920

# MARKET SL close:
fill_price = sl_level * (1 - SLIPPAGE[coin] / 100)
# slightly worse than SL level

# Candle closes at 14:30:00
# WS tick detected at 14:30:01
# Real fill = next bid/ask, not close price
# Simulate: sl_fill = close_price * (1 - 0.05%)

Profit Settlement Model — FINAL

How it works

User deposits on their own exchange. Gives trade-only API key. Funds never leave their account — no custody. Agent trades. Platform collects 20% of profit every 15 days.


Cycle 1 — 120% threshold trigger

User deposits $50
Trigger at $110 (120% growth)
Profit $60
Platform fee (20%) $12
Remaining after fee $98

User gets 2 options at settlement

Option What happens Still trading
A — Withdraw capital User gets $50 back. $48 (80% profit) stays trading. $48
B — Compound Nothing withdrawn. Full $98 keeps trading. $98

Cycles 2+ — every 15 days from here

  • Every 15 days: calculate profit since last settlement
  • Platform takes 20% of that profit
  • If no profit in the window → no fee, cycle resets automatically
  • Fee not paid within 48hrs → trading paused until settled
  • Fee paid → trading resumes within 5 minutes

Compounding snowball example

Cycle Opening Closing Profit Platform 20%
1 (trigger) $50 $110 $60 $12
2 (day 15) $98 $120 $22 $4.40
3 (day 30) $138 $163 $25 $5

Key rules:

  • 20% always taken on profit — never on capital
  • No profit in a cycle = no fee
  • Funds stay on user's exchange — no custody risk

Mobile App — Distribution Strategy

Stack: Expo + React Native

Single codebase. Builds to iOS, Android, and web simultaneously. No separate teams or repos. Ship web first, mobile stores when ready.


Launch order:

  • Phase 1 — Website download first — Host the APK (Android) and IPA (iOS sideload) directly on tradelikeme.xyz. Users download from the website. This lets us ship fast, iterate without App Store review delays, and validate with real users before committing to store policies.
  • Phase 2 — App Store / Play Store — Once the app is stable, win rate is proven with live users, and we're confident in the UX, submit to Google Play and Apple App Store. This is the credibility unlock that drives organic installs.

Why this order makes sense:

  • App Store review takes 1–7 days and can reject for any reason — no blockers during early testing
  • APK sideload on Android is trivial. iOS requires TestFlight or sideloading (slightly more friction but doable for early users)
  • Expo EAS Build handles both — one command builds the APK/IPA
  • When we do submit to stores, the codebase is already production-tested — lower rejection risk

One codebase = one place to fix bugs, one deployment, web + mobile always in sync.

🧠 Agent Architecture (Analysis + Execution)
Analysis Layer — Single Brain
  • AA1
    Build analysis engine — single process that scans all instruments, runs all gates, calls Claude Opus on zone-touch events. Runs once for all users regardless of user count.
  • AA2
    Build signal queue — when analysis finds a valid setup, publish it to a shared DB table or Redis queue: symbol, direction, entry zone, TP1, TP2, SL, timestamp
  • AA3
    Signal expiry — mark signals stale after 30 minutes. Executor agents ignore expired signals — don't enter a trade on old data.
Execution Layer — User Agent Swarm
  • AA4
    Build user executor — lightweight agent per user. Subscribes to signal queue, reads user's risk mode, sizes the trade, places it on user's exchange. No scanning, no AI.
  • AA5
    Per-user isolation — each executor manages only that user's trades, TP/SL, body-close SL, and settlement. One user's exchange error never affects another user.
  • AA6
    Trading hold gate — executor checks user's payment status before acting on any signal. If on hold → skip all signals until hold lifted.
  • AA7
    Auto-spawn on signup — when new user connects their exchange API key, spin up their executor automatically. No manual deployment needed.
  • AA8
    Auto-stop on unsubscribe — when user disconnects or deletes account, shut down their executor cleanly, close any open positions first.
Health Monitoring
  • AA9
    Analysis engine heartbeat — ping every 5 minutes. If silent for 10 minutes → Telegram alert to admin immediately.
  • AA10
    Executor heartbeat — each user executor pings every 5 minutes. If any executor goes silent → alert admin + alert that user via Telegram.
  • AA11
    Auto-restart — if analysis engine or any executor crashes, Docker auto-restarts it within 30 seconds. Log the crash reason to DB.
